  74      ''' 
  75      FileSequenceChecker 
  76      =================== 
  77       
  78      A class for (recursively) checking a directory with file sequences for missing files. 
  79       
  80      Modus Operandi 
  81      -------------- 
  82       
  83      Processing happens in three steps: 
  84       
  85          1. Generate a directory contents dict by splitting each file name  
  86             into a name and sequence number part when generating lists  
  87             for each file sequence present in the dir.  
  88             We sort these lists in a natural way, first by sequence number,  
  89             then by file name. This step is also responsible for including  
  90             or excluding files based on criteria such as command line args  
  91             (C{--include/--exclude}) and/or L{FILEEXCLUDES}.  
  92              
  93             For each directory a file name list is saved, containing dicts 
  94             with 5 keys: 
  95              
  96                 1. file name (C{filename}),  
  97                 2. sequence number (C{seqnum}),  
  98                 3. file extension (incl. dot) (C{fileext}),  
  99                 4. order (C{order}), 
 100                 5. file name, 2nd part (optional) (C{filename2}) 
 101                  
 102             The order (C{'normal'} or C{'reverse'}) specifies which comes first:  
 103             file name or sequence number. 
 104              
 105             A fifth key C{filename2} may exist which contains the second filename  
 106             part. If present this part will be re-inserted again (appropriate to 
 107             the order) when constructing the missing file names as part of creating 
 108             a missing files series.   
 109              
 110             The order must be stored explicitly to be able to print missing  
 111             file names in a way directly relating to the first file in the  
 112             sequence.  
 113              
 114             It is also worth noting that L{self.start} and L{self.end} do B{NOT} play  
 115             an important role here. The constraining to files within a particular  
 116             start and end interval is utilized and verified later within the  
 117             comparing function. 
 118              
 119          2. Iterate over the prepared directory contents and from each file name list 
 120             pass dicts representing current file name parts and next file name parts  
 121             to a comparing function.  
 122              
 123             By comparing clues from both dictionaries we can determine if files are  
 124             missing and how many. Lists with missing file names are created on 
 125             a per directory basis. Each list is then added to a dictionary, keyed  
 126             by path to the directory that's currently processed. 
 127              
 128          3. L{self.processdir()} returns this dictionary, or an empty dictionary  
 129             if no there are no missing files, so that the caller can further do her 
 130             own processing. 
 131              
 132      Caveats 
 133      ------- 
 134       
 135      Even after careful testing and development, there are still some caveats left. 
 136      These are mostly coming from not being able to do everything automatically and  
 137      then hide it away as implementation detail, especially when faced with inherent 
 138      ambiguities.  
 139       
 140      For example: certain file names are just not distuingishable enough to match one 
 141      default regex pattern over the other. This is because from a regex based matching  
 142      perspective strings like C{v105-name} and C{name-v105} are just too similar.  
 143      Think about it: to match the first string you could say: match any lowercase letter  
 144      followed by a number.  
 145       
 146      But can you be sure that you will always need to match just one lowercase letter?  
 147      I would argue you can't. Then you say: match any lowercase letter any number of  
 148      times followed by a number. If you now compare this with the second string you  
 149      will find that this definition will match both strings. Of course you can mess 
 150      around with "rooting" to different sides of the string (C{^} vs. C{$}) and also  
 151      impose a superficial ordering of less-likely to likely when applying different  
 152      patterns, but fundamentally there will always be a at least one case that is  
 153      too ambiguous compared to another case. 
 154       
 155      Ambiguities like these are hard to resolve by automation. Any good API will defer  
 156      these decisions to the user of the class and thus delegate the choice to the one  
 157      who should know most about specific needs. 
 158       
 159      In the case of the CLI that L{FileSequenceChecker} was originally made for,  
 160      command line arguments give the end user the ability to define a pattern and  
 161      a replacement template themselves, so as to cater to these special cases.  
 162   
 163      Example Use 
 164      ----------- 
 165

 168      sequence numbers within the range 0 through 10.

 210      -------------------- 
 211       
 212      As this class has I{Sequence} in its name, a strong case was made

 219      objects like lists, tuples etc.

 228      list of file sequence candidates for processing later on.
